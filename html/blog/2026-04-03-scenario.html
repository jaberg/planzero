    Simulation Framework

    PlanZero's simulation framework involves a few types of thing:

    • time series objects, which contain the data of time-varying quantities
    • dynamic element objects, which implement rules for the evolution of time series
    • scenario objects, which are sets of time series and dynamic elements.
    This section introduces these object types, and later sections explain how they are used for modelling.

    Time Series Objects

    A time series object (sometimes written as a TimeSeries) represents a time-varying quantity. PlanZero posts have used TimeSeries objects already, to represent e.g. the various sectoral emissions of greenhouse gases for the NIR. The key attributes of a TimeSeries are:

    • a set of what numeric values the series takes
    • a set of times marking when the series takes these values
    • a unit of measure for the values
    • an interpolation mode, indicating how to interpret the times and values.
    A TimeSeries is a bit like two matched columns in a spreadsheet: one column being the times, and the other being the values. Relative to columns in a spreadsheet, a TimeSeries restricts all of its values to be in a single common unit of measure, and imposes querying semantics on the TimeSeries according to the interpolation mode.

    The main purpose of PlanZero's TimeSeries objects is to support querying the value at particular times, so that XXX. The querying logic in PlanZero's TimeSeries objects depends on the interpolation mode. As of writing, there are two interpolation modes:

    • No interpolation: the series is only defined at the times listed explicitly, at which times the value is the corresponding value.
    • Current interpolation: the series is defined at all times, to be the most-recent value prior to (or coincident with) time t.
    The "No interpolation" mode is suitable for series such as an annual CO2e emission in the NIR. In that kind of time series, there are amounts for e.g. years, but there is no implied annual total for dates in between; the annual totals are only meant to apply to whole years. The "Current interpolation" mode is suitable for other kinds of time series, such as e.g. the number of cattle in Canada. The nature of that time-varying quantity is different in that the question of "how many cattle are there in Canada?" can be reasonably asked of any date, regardless of what (time, value) pairs are being used to implement a TimeSeries approximating that time-varying quantity. Time-varying quantities such as emission factors, prices, and populations are best-represented as "Current interpolation" TimeSeries objects. Time-varying quantities such as quarterly or annual summaries are best-represented as "No interpolation".

    The purpose of the units of measure and the interpolation modes in TimeSeries objects is to prevent logic errors in modelling. When adding TimeSeries using compatible units (e.g. pounds and kilograms), unit conversion is automatic so that there's no chance of using an incorrect conversion factor by mistake. When adding TimeSeries using incompatible units (e.g. kilograms and kilograms per year), the library code raises an exception to indicate that there's an error in the model that needs correcting. Similarly, the intention of the interpolation modes is to catch logical errors. For example, multiplying annual greenhouse gas totals (whose mode would be "no interpolation") by emission factors (with mode "current interpolation") would result in a timeseries with a mode of "no interpolation"; attempting to query that timeseries at arbitrary times would raise an exception because it's likely a mistaken usage.

    The next type of object in the simulation framework is the dynamic element. Dynamic elements are bits of modelling logic in the PlanZero software that define TimeSeries objects.

    Is a rule for determining the values of some TimeSeries objects (outputs), based on some modelling parameters, and the values of other TimeSeries objects (inputs). There are several styles of Dynamic Element in PlanZero: some simply import data from StatsCan tables the X DynamicElement implements some simple geophysics the Y DynamicElement implements a hypothetical government policy and the Z DynamicElement implements hypothetical economic behaviour (how farmers might adopt Bovaer).

    DynamicElement objects declare the names of the TimeSeries they require as input and the names of the TimeSeries they define as outputs. The simulation engine uses this information to ensure the compatibility of all of the DynamicElement objects that make up a simulation.

    There is support for circular dependencies, because DynamicElements must specifiy, for each input, whether the exactly-current value of that input is required, or any recent-enough value will do. Circular dependencies through exactly-current required inputs are disallowed, but usually these circular dependencies can be resolved by configuring one of the DyanamicElement objects to accept an almost-exactly-current value.

    and Dynamic Element objects, that represent the rules for how those values should change. Is a data structure of values of something at a sequence of times. The value at a particular time can be undefined, or a number or a random variable. The whole time series generally has a unit, so the series represents e.g. a sequence of Mt {{CO2e|safe}} or Canadian dollars. See ureg.py for the set of units used in PlanZero.

    Using the Framework

    To date, PlanZero posts have talked about critical success factors, barriers, and strategies informally. I will make an effort, going forward, to reflect at least some of these as code objects.

    CSFs

    These are names for TimeSeries objects, and sometimes DynamicElements that maintain a relationship between a few named TimeSeries objects. For example, a post may say name a strategy of "increasing engine efficiency". Going forward, I'll attempt to formalize that by stating the name of this efficiency variable. In the case of efficiency variables specifically, I'll also name the numerator and denominator, and the DynamicElement that relates all three.

    Barriers

    What I've been calling barriers in posts to date typically correspond to constraints. In PlanZero, these constraints are implemented as TimeSeries objects that take a value of 0 for times when the constraint is not violated, and another value when the constraint is violated. Generally a DynamicElement is required to determine whether each constraint is violated.

    Strategies

    Strategies are DynamicElement and TimeSeries objects that implement modelling assumptions, and serve to provide values for the TimeSeries objects that are the inputs to the CSF and Barrier DynamicElements.

    Not all Strategies are well-enough developed to be included in a Scenario. PlanZero includes Strategies that are quite speculative: all that's required to create a Strategy in PlanZero is a brief description of the idea, and a URL to learn more about it. In order to be eligible to participate in a scenario, a Strategy must be fully fledged as a DynamicElement: it must declare which TimeSeries variables it consults as inputs, and which ones it defines as outputs, and what logic it uses to relate them.

    Scenario

    A Scenario Is a data structure that comprises all of the TimeSeries objects, and all of the DynamicElement objects that make up a model of Canada's future, with regards to emissions.
