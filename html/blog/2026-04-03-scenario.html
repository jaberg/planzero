    Introduction

    This post represents a break from the series replicating 2025 National Inventory Report (NIR) sectoral totals. It introduces strategies and scenarios as formal elements of the PlanZero modelling framework, by way of first examples of Bovaer usage (as a strategy) and scaling-of-current-products (as a scenario, aka "scaling"). The PlanZero site now includes top-level tabs for both types of entity:

    • Strategies — code elements representing things that could be done
    • Scenarios — simulations of strategies, and their consequences
    That said, PlanZero is just getting started. For anyone that would like to see a more-developed pathways-to-net-zero tracker for Canada, check out 440 Megatones Pathways Tracker, and the research behind it. It's great work and I intend to replicate it to understand it better.

    The post is organized as follows:

    • The Simulation Framework section introduces how PlanZero simulates things like an enormous spreadsheet with time series objects (columns of numbers) and dynamic element objects (initialization and formulas).
    • The Scenario Modelling section explains how the concepts of critical success factors, barriers, and strategies can be (a) formalized in terms of time series and dynamic elements, and (b) combined to form scenarios.
    • The Scaling Scenario, Starting with Bovaer section documents the dynamic elements of the new scaling scenario, including the strategy of scaling the usage of Bovaer (and its associated barriers).

    The post is maybe the most technical so far (or most theoretical?), but I hope it might offer a design introduction to anyone interested in using this modelling framework. I intend to follow up on this post with others that add strategies and fill out the scaling scenario; with the technical foundation here, these future posts can be much shorter.

    Simulation Framework

    So far the posts have mostly been about trying to re-create data in the National Greenhouse Gas Inventory Report from other data sources, which I was doing to get started. The vision for the PlanZero project is that it will extend those sectoral emissions into the future with models that project how emissions might change in future. This is the first post that does that kind of projection. This section introduces the simulation engine used in PlanZero to run models that work a bit like a large spreadsheet. PlanZero's simulation framework involves two concepts:

    • time series objects, which contain the data of time-varying quantities (such as e.g. temperature, or annual emissions)
    • dynamic element objects, which implement rules for the evolution of time series data
    The next two subsections explain each of these in more detail, and the next section (Scenario Modelling) explains how these are used to define scenarios.

    Time Series Objects

    A time series represents a time-varying quantity in some unit of measure. PlanZero posts have used time series objects already to represent just about everything that was plotted as a function of time, e.g. both the various sectoral emissions of greenhouse gases in the NIR, and the attempts to reconstruct them. The key attributes of a time series are:

    • a set of what numeric values the series takes
    • a set of times marking when the series takes these values
    • a unit of measure for the values
    • an interpolation mode, indicating how to interpret the times and values.
    A TimeSeries is a bit like two matched columns in a spreadsheet: one column being the times, and the other being the values. Relative to columns in a spreadsheet, a time series object restricts all of its values to be in a single common unit of measure, and imposes querying semantics on the TimeSeries according to the interpolation mode.

    Time series are important in PlanZero because they are the basis of simulation. PlanZero simulates scenarios by repeatedly figuring out what the next value in each time series should be. The figuring-out of the next value is done by dynamic elements, which we'll talk about in the next sub-section.

    Before we get to dynamic elements though - a brief defence for making a custom data type with units and an interpolation mode. Unit conversion is a notorious source of error in systems that are difficult to test. One recent high-profile example is the loss of the Mars Climate Orbiter in 1999. Time series objects in PlanZero are always in some unit of measure to try to avoid such errors, and PlanZero has introduced additional units of measure (cattle, CAD, kg-of-CO2) to reduce the likelihood of combining numbers inappropriately. A second defence against modelling error is the use of an "interpolation mode" on time series. There are currently two modes: the value of the series at points in time that aren't explicitly represented is either (a) undefined, or (b) defined to be the most-recent value. Series with different modes can be combined, but bearing these semantics in mind has helped catch modelling errors. These mechanisms don't guarantee a model is good, but they do help ensure that a simulation result reflects the intended modelling assumptions.

    Dynamic Element Objects

    Whereas time series objects are tables of numbers like what you can see in spreadsheet columns, PlanZero's dynamic element objects are more like the formulas that define spreadsheet cells. Dynamic elements define how time series objects should be initialized, and how they should be extended by iterative computation. A dynamic element object is like a spreadsheet of a specific form:

    • a single shared set of times runs down the first column
    • each other column has the numeric values of a time series
    • the initial rows (earlier times) are initialized using custom logic
    • the subsequent rows (later times) are computed recursively, each row from previous ones, always re-applying the same formula
    The dynamic elements that define a scenario can read each others' columns, but they can only write their own columns.

    Dynamic elements are important to know about, because they encode the modelling assumptions that make up a PlanZero scenario. PlanZero simulation engine simulates a scenario by using the dynamic elements in a scenario to update the rows of the various time series in chronological order, and in order of dependency when there are ties in the chronological ordering. The next section explains how critical success factors, barriers, strategies, and scenarios can be formalized as time series and dynamic elements.

    Scenario Modelling

    The purpose of PlanZero is to model future scenarios, and to explore strategies for how Canada might achieve a net-zero economy. I think of this modelling as being a combination of strategic management and numerical optimization. Strategic management, to quote wikipedia, "involves the formulation and implementation of the major goals and initiatives taken by an organization's managers on behalf of stakeholders, based on consideration of resources and an assessment of the internal and external environments in which the organization operates." If we consider Canada as an organization, the federal government has, among it's many goals and initiatives, set a goal to achieve net-zero by 2050 (earlier post, government site). As explained in the following sections, a framework used in strategic management helps formalize this goal in terms of a few types of time series and dynamic element, and numeric optimization can be used to search over strategies for desirable predicted futures. I hope that invoking the terminology of goals, critical success factors, barriers, and strategies helps to make the modelling more accessible and engaging, and thereby solicits a greater quantity of constructive feedback.

    The word "goal" has a common-sense definition, but in the context of governmental planning and legislation, it is also appropriate to consider the term within the realm of strategic management. Strategic management does not offer a single definition of what a goal is, or a canonical articulation of what good management looks like. I have an extremely limited knowledge of that field, but for what it's worth, the planning process of the federal government brings to my mind the framework articulated in "The Executive's Guide to Facilitating Strategy" by Michael Wilkinson (which I'll refer to as EGFS). In the terminology of the EGFS, an organization's SMART goals (Specific, Measurable, Achievable, Relevant, and Time-bound) should be broken down systematically during an implementation planning process. The goal of net-zero by 2050 is questionably "Achievable" but otherwise qualifies as a SMART goal. EGFS suggests breaking down such a goal into three sets of considerations and hypotheses, before developing detailed implementation plans for specific strategies:

    • Critical Success Factors are key conditions that must be created to achieve one or more objectives.
    • Barriers are existing or potential challenges that hinder the achievement of one or more objectives.
    • Strategies are broad activities required to achieve a goal, create a critical condition, or overcome a barrier.
    PlanZero posts focused on replicating NIR emissions estimates (such as "Enteric Fermentation: Emissions Calculations") have included sections with these names, and informal thoughts aligned with these definitions. The scenario modelling introduced in this post involves formalizing these notions as code objects that can be simulated and compared numerically to determine if, e.g. a particular strategy is a good idea.

    The next subsections explain how the EGFS concepts of critical success factor, barrier, and strategy can be formalized in terms of time series and dynamic elements, and used in that way to structure and interpret scenario simulations for the purpose of evaluating strategies.

    Formalizing Critical Success Factors

    Critical success factors are introduced in EGFS as "key conditions that must be created to achieve one or more objectives." In PlanZero, the objective is, for each IPCC sector, to reduce emissions as much as possible, even producing negative emissions when that's possible. Therefore PlanZero's success factors can be defined for each IPCC sector to reflect the numeric drivers of emissions in that sector. For example, in the case of enteric fermentation, the emissions calculation (see this earlier post approximating the enteric fermentation emissions) could be articulated as the product of two terms (ignoring ruminants other than cattle): number of cattle, and average methane emitted per head of cattle. These two terms suggest two critical success factors:

    • Minimize the number of cattle.
    • Minimize the average amount of methane emitted per head of cattle.
    Minimizing either or both of these terms would, mathematically, reduce enteric fermentation emissions. Without minimizing one or the other, enteric fermentation emissions will not be reduced. Therefore, it's fair to say these are two "key conditions that must be created to achieve one or more objectives" of emissions reduction in this sector.

    The two metrics mentioned in the critical success factors (number of cattle, and average amount of methane emitted) are also time series in PlanZero, because that's how I computed the NIR estimate for that sector. This has, in fact, been my motive in replicating the NIR estimates. I think the EGFS framework is useful, and I wanted to find critical success factors for each of the sectors. I believe these, at least in the context of PlanZero, can be formalized in terms of a triplet of (a) a sector, (b) an optimal value, and (c) a TimeSeries object, which can be called a Key Performance Indicators or KPI. This triplet of information is attached, in PlanZero, to special dynamic elements which carry the following information pertaining to a set of critical success factors:

    • the names of the KPI time series variables
    • whether each one should ideally be maximized, minimized, or maintained at a specific value
    • which IPCC sector(s) are affected by the critical success factors
    Scenarios will be discussed below, but the critical success factors identified by these dynamic elements serve as default indicators to characterize each IPCC sector when visualizing scenario simulations.

    Formalizing Barriers and Strategies

    Barriers are introduced in EGFS as "existing or potential challenges that hinder the achievement of one or more objectives" and strategies as "broad activities required to achieve a goal, create a critical condition, or overcome a barrier." In PlanZero, these two concepts are formalized hand-in-hand: EGFS strategies are formalized as relatively simple dynamic elements that reflect an approach, such as "feed as many cattle Bovaer as possible"; and EGFS barriers are formalized as all of the dynamic elements required to model the effect of such a strategy on critical success factors, such as "how many cattle can be fed Bovaer", "what would it cost", and "how much good would it do?"

    With both barriers and strategies being formalized as dynamic elements that are moreover developed "hand-in-hand", it's worth clarifying in terms of dynamic elements where one ends and the other begins. For greater clarity, a PlanZero Strategy should be an optional dynamic element that puts a trajectory in motion, and can be omitted to mean "not doing it". Omitting a strategy should not compromise the validity of a scenario simulation. In contrast, PlanZero barriers implement logic that is integral to the validity of the scenario; with regards to enteric fermentation for example, cattle produce methane at certain rates (which depend on e.g. Bovaer consumption) regardless of whether any cattle are fed Bovaer. The logic and assumptions that PlanZero uses to estimate methane emissions from cattle is implemented in a dynamic element, and that dynamic element is a barrier. This logic should always be included in simulation in order to estimate emissions for the sector.

    Barrier dynamic elements model

    • how fast strategies will take effect
    • what costs / profits are expected
    • what will happen in the meantime before / as they work
    Strategy dynamic elements model
    • when a strategy should start
    • how to use investment (if applicable)
    Barrier objects in PlanZero have the job of predicting the future. I hope, over time, to build up and refine the library of barriers in PlanZero, because I believe that this library will become the most valuable component of the project. Strategy objects are meant to reflect policy changes or product / project development efforts that could be undertaken; my vision is, at some point, to apply numerical optimization algorithms to tune the timing and degree of investment in strategies, but for now they are simply included all-or-nothing.

    Scenario Simulation

    Scenarios have been mentioned several times in this post already, but this subsection explains what I've meant in more detail. In the terms of the previous section (Simulation Framework) a PlanZero scenario is a set of dynamic elements, and the time series objects they define. In the terms of this section (Scenario Modelling) a PlanZero scenario is a set of critical success factors, barriers, and strategy elements, as well as a few built-in dynamic elements that, together, are capable of simulating the evolution of the emissions sectors used in Canada's 2025 NIR. The built-in dynamic elements are capable of simulating future emissions even without any critical success factor, barrier, or strategy elements:

    • Other NIR Historic Actuals defines time series objects for IPCC-sector emissions that haven't been modelled by other dynamic elements. It initializes them with values from the NIR, and projects that they remain constant in the future.
    • Atmospheric Chemistry combines sectoral GHG emissions into sectoral and national {{CO2e|safe}} totals, and simulates the effect on a simple planetary heat model.
    • Subsidy Accounting totals any subsidy requirements across barriers. At some point this may include estimated impacts in e.g. healthcare, insurance, climate mitigation projects, and disaster relief; currently it does not.
    Scenario simulation is probably best illustrated by example. Without further ado, let's continue to the next section, which describes PlanZero's first scenario: a simulation of possible futures with and without scaling the use of the bovine feed additive Bovaer.

    The Scaling Scenario, Starting with Bovaer

    Bovaer a brand name for a novel feed additive for cattle and sheep, also known as 3-Nitrooxypropanol, also known as 3-NOP. It is a synthetic organic compound that, when added to ruminant livestock feed in the correct manner, has the effect of inhibiting methane production in their first stomach. Bovaer is approved for use in Canada. This section explains how PlanZero can simulate aspects of Canada's future to estimate the consequences of using Bovaer, and of not using it. This post does not establish a position on the question of whether Canada should fund Bovaer for a number of reasons, foremost among them being that Bovaer is the only strategy in the model (and running a close second would be that the scenario makes no attempt to model government decision-making). What the post does do, is estimate that Bovaer costs about $175/t{{CO2e|safe}} according to the modelling assumptions in the barriers below.

    Critical Success Factors and Strategies for Enteric Fermentation Emissions

    We've already seen the currently-modelled critical success factors and strategies for enteric fermentation in the examples given above. As of writing they are code objects, and rendered automatically as part of the new scaling scenario page: critical success factors, strategies. Reproducing them here, as that page is expected to change in future:

    Strategies

    SectorStrategyDescription
    Enteric Fermentation, Other Product Manufacture and Use ScaleBovaer Use as much Bovaer as farmers will take

    Critical Success Factors

    SectorMaximize / Minimize / MaintainTimeSeries KPI
    Enteric Fermentation Minimize bovine_methane_per_head
    Enteric Fermentation Minimize bovine_headcount

    The strategy of using as much Bovaer as farmers are prepared to take is called "ScaleBovaer" and it is defined, as of writing, by its Python language implementation to work by setting elements of the time series called "bovine_population_fraction_on_bovaer" to equal those of another time series called "max_fraction_of_cattle_on_bovaer". The KPIs for enteric fermentation emission reduction are called "bovine_headcount" and "bovine_methane_per_head". These time series are related by the barrier elements described in the next section.

    Barriers

    We've seen some of the currently-modelled barriers for enteric fermentation emissions in the examples given above, but not all of them. As of writing, the following are code objects, and rendered automatically in the Barriers section of the new scaling scenario page. Reproducing them here (as that page is expected to change in future):

    Barriers

    SectorBarrierDescription
    Enteric Fermentation Bovaer Adoption Limit Assume Bovaer will only be adopted by, at most, 5.0 percent / year of cattle operations
    Enteric Fermentation, Other Product Manufacture and Use Bovine Population Model cattle population, production of methane (considering Bovaer), milk and beef (TODO)
    Enteric Fermentation Bovaer Monitoring Assume monitoring Bovaer usage costs 1000.0 CAD / farm / year for paperwork and 3000.0 CAD / farm / year for on-site inspection

    The "Bovaer Adoption Limit" barrier defines the "max_fraction_of_cattle_on_bovaer" time series by implementing an assumption that whenever Bovaer starts being completely subsidized by e.g. the government, that farmers might adopt its usage at a rate of at most 5% of all farmers per year. Further, it assumes that organic farmers (estimated to be 1% of farmers) will never adopt Bovaer, because Bovaer is incompatible with the production of organic milk and beef. The details of the Python implementation as of writing are available on github.

    The "Bovine Population" barrier defines a large number of time series: all of the cattle headcounts per cattle type (calves, beef vs. dairy vs. slaughter heifers, beef vs. dairy cows, bulls), per province and territory, and per whether they are on Bovaer or not. It estimates the methane produced by each of these sub-populations of cattle, and adds them up to get a total. Cattle on Bovaer are modelled as producing 30% less if they are dairy cows, 45% less if they are beef cows, and similar numbers that I guessed if they are other kinds of cattle (I could not find data). It also estimates the cost of the Bovaer required — from $0.20/day for calves to $0.50/day for adults — and registers this cost as requiring a subsidy. It also estimates the {{CO2e|safe}} required to produce Bovaer; I couldn't find a reference for this number, but I'm guessing it is about 5% of the greenhouse gas reduction delivered by the Bovaer usage. The {{CO2e|safe}} is attributed to the IPCC Sector "Industrial / Other Product Manufacture and Use" as {{CO2|safe}} until I have a better model of how Bovaer is produced. The details of the Python implementation as of writing are available on github.

    The "Bovaer Monitoring" barrier defines a single time series called "bovaer_monitoring_cost_annual" that is registered as requiring a subsidy. The subsidy amount is calculated based on guesses of $1000 / farm /year for administrative overhead, and $3000 / farm / year for onsite inspection. Confidence intervals are not yet supported in PlanZero, but if they were, I would use large ones around these figures as I did not find references (github issue here). This barrier uses a rough estimate of 160 cattle / farm on average, rather than the actual farm data available from Statistics Canada which has hovered around 160 in recent years. The details of the Python implementation as of writing are available on github.

    Simulating Canada's Future with Bovaer Usage

    The combination of all of the critical success factors, barriers, and strategies is a set of 9 dynamic elements (3 built-in, 2 critical success factor, 1 strategy, and 3 barriers), and 1267 time series objects (the ones named above, plus many others, such as emissions of each GHG for each province and territory, for each of 71 IPCC sectors, and per-province per-cattle-type headcounts). The whole set is listed on the scaling scenario page (see sections on dynamic elements and time series), although the set of elements listed there will grow as more barriers and strategies are added in future. Running this simulation for 100 years lets us visualize a future trajectory of emissions according to the IPCC sectors currently used in the NIR.

    For most sectors, the figure shows historical actuals until 2023, followed by constant projections as expected when using the "Other NIR Historic Actuals" dynamic element. For the "Enteric Fermentation" sector (scaling scenario), we see a steady reduction in emissions as our simulated farmers adopt Bovaer at 5% / year. At the same time, we see a steady rise in emissions in the "Other Product Manufacture and Use" sector. After all eligible farms have adopted Bovaer annual emissions are reduced by 9.5 Mt {{CO2e|safe}}. The magnitude of subsidy required is shown in dark red, and plotted against the right vertical axis, in billions of dollars. As farmers adopt Bovaer it rises to a total just over 1.6 billion dollars per year. Since Bovaer usage and cost is proportional and synchronous, the cost of emissions reduction from using Bovaer can be computed directly by dividing these two numbers: 1.6 billion dollars / 9.5 Mt {{CO2e|safe}} equates to $175.36 / t{{CO2e|safe}}. Ablative analysis (looking at the results by comparing with the removal of each strategy one at a time) is available for the ScaleBovaer.

    Conclusion

    This post has introduced PlanZero's simulation framework in terms of computational elements (time series and dynamic elements) and in terms of strategic management concepts (critical success factors, barriers, strategies, scenarios). The post brings two new tabs to the top of the PlanZero website: strategies lists the strategies implemented in PlanZero, which scenarios include them, and which IPCC Sectors they impact; scenarios lists the scenarios that are implemented in the software. Currently, there is just one strategy (ScaleBovaer) and one scenario (scaling) but it is my intention to add more. I would like to get into a cadence of alternating between NIR-sector posts and scaling-strategy posts. I imagine, over time, that PlanZero may grow to include a small number of standard scenarios (perhaps "do nothing", "no-subsidies", "what about counting on a few speculative technologies...") but I'd concede I'm a bit hazy on how scenarios will be used, so I won't worry about it yet. More urgently than scenarios, PlanZero needs strategies (and associated critical success factors and barriers), starting with products and technologies that are closest to mass adoption. For anyone that would like to see a more-developed pathways-to-net-zero tracker for Canada, check out 440 Megatones Pathways Tracker, and the research behind it. It's great work and I intend to replicate it to understand it better (github issue). The next strategy posts, in no particular order, should probably cover at least some of the following (the links in the list are to github issues reminding me to implement them):

    • Solar, wind, tidal, and/or nuclear energy for grid-scale power generation
    • EVs for light-duty vehicles
    • Heat Pumps for residential and commercial/institutional heating

    Until next time,
