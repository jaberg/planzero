    Simulation Framework

    PlanZero's simulation framework involves a few types of thing:

    • time series objects, which contain the data of time-varying quantities
    • dynamic element objects, which implement rules for the evolution of time series
    • scenario objects, which are sets of time series and dynamic elements.
    This section introduces these object types, and later sections explain how they are used for modelling.

    Time Series Objects

    A time series object (sometimes written as a TimeSeries) represents a time-varying quantity. PlanZero posts have used TimeSeries objects already, to represent e.g. the various sectoral emissions of greenhouse gases for the NIR. The key attributes of a TimeSeries are:

    • a set of what numeric values the series takes
    • a set of times marking when the series takes these values
    • a unit of measure for the values
    • an interpolation mode, indicating how to interpret the times and values.
    A TimeSeries is a bit like two matched columns in a spreadsheet: one column being the times, and the other being the values. Relative to columns in a spreadsheet, a TimeSeries restricts all of its values to be in a single common unit of measure, and imposes querying semantics on the TimeSeries according to the interpolation mode.

    The main purpose of PlanZero's TimeSeries objects is to support querying the value at particular times, in order to compute the values of other time-varying quantities required to simulate scenarios. The querying logic in PlanZero's TimeSeries objects depends on the interpolation mode. As of writing, there are two interpolation modes:

    • No interpolation: the series is only defined at the times listed explicitly, at which times the value is the corresponding value.
    • Current interpolation: the series is defined at all times, to be the most-recent value prior to (or coincident with) time t.
    The "No interpolation" mode is suitable for series such as an annual CO2e emission in the NIR. In that kind of time series, there are amounts for e.g. years, but there is no implied annual total for dates in between; the annual totals are only meant to apply to whole years. The "Current interpolation" mode is suitable for other kinds of time series, such as e.g. the number of cattle in Canada. The nature of that time-varying quantity is different in that the question of "how many cattle are there in Canada?" can be reasonably asked of any date, regardless of what (time, value) pairs are being used to implement a TimeSeries approximating that time-varying quantity. Time-varying quantities such as emission factors, prices, and populations are best-represented as "Current interpolation" TimeSeries objects. Time-varying quantities such as quarterly or annual summaries are best-represented as "No interpolation".

    The purpose of the units of measure and the interpolation modes in TimeSeries objects is to prevent logic errors in modelling. When adding TimeSeries using compatible units (e.g. pounds and kilograms), unit conversion is automatic so that there's no chance of using an incorrect conversion factor by mistake. When adding TimeSeries using incompatible units (e.g. kilograms and kilograms per year), the library code raises an exception to indicate that there's an error in the model that needs correcting. Similarly, the intention of the interpolation modes is to catch logical errors. For example, multiplying annual greenhouse gas totals (whose mode would be "no interpolation") by emission factors (with mode "current interpolation") would result in a timeseries with a mode of "no interpolation"; attempting to query that timeseries at arbitrary times would raise an exception because it's likely a mistaken usage.

    There are 1262 time series objects involved in the current scaling scenario. Most of the time series objects are indexed by year, although the heads-of-cattle counts are indexed by half-years because statistics canada reports headcounts at January 1 and July 1. There isn't a single obvious way to group, or organize all of these time series objects; techniques have been developed to help analysts explore such data, but researching, selecting, and integrating products for data visualization with PlanZero is left to future work. Broadly speaking, I think of the timeseries in the scenario so far as follows:

    • cattle headcounts, for each type of cattle, and for each province
    • emissions of each greenhouse gas, for each IPCC category
    • atmospheric timeseries related to energy forcing as described in this earlier post
    • Bovaer-related timeseries, such as how many cattle are on it, and how many farmers might adopt it
    • costs requiring subsidy: e.g. Bovaer purchase and monitoring
    The full table of timeseries values is shown on the scenario page .

    Dynamic Element Objects

    Whereas TimeSeries objects are just tables of numbers like what you can see in spreadsheet columns, PlanZero's DynamicElement objects provide something like the formulas that sometimes generate those numbers. Dynamic elements define what the TimeSeries objects should be, and how they relate to each other. A DynamicElement object is like a spreadsheet of a specific form:

    • a single shared set of times runs down the first column
    • each other column has the numeric values of a TimeSeries object
    • the initial rows are initialized using custom logic
    • the subsequent rows are computed recursively, each row from previous ones in the same way for each row
    The DynamicElements that define a scenario can read each others' columns, but they can only write their own columns. The PlanZero simulation engine simulates a scenario by updating the rows of the various spreadsheets in chronological order, and in order of dependency when there are ties in the chronological ordering.

    By construction, are no more DynamicElement objects than TimeSeries. In practice, there are far fewer. As of writing, the Scaling scenario has over a thousand time series, but it has just eight dynamic elements (see also the current scaling scenario dynamic elements):
    Index Dynamic Element Name Role Description
    1 BovaerAdoptionLimit barrier Assume that no more than 5.0 percent / year of farmers will switch to administering Bovaer in any given year, but that adoption can ultimately go all the way to 100%.
    2 BovinePopulation barrier Assume cattle produce methane (although less-so if they are fed Bovaer) and they also produce milk and beef. The evolution of the cattle population is projected to remain constant.
    3 BovaerMonitoring barrier Assume paperwork and onsite monitoring costs to certify emission reductions.
    4 EntericFermentation CSFs CSF All formalized CSFs related to Enteric Fermentation.
    5 ScaleBovaer strategy Use as much Bovaer as farmers will take
    6 Other NIR Historical Actuals built-in Populate otherwise-missing IPCC Categories with historical actuals from NIR-2025
    7 AtmosphericChemistry built-in Combine GHG emissions into a CO2e estimate using GWP-100 emission factors, and also simulate a simple radiative forcing and planetary heating model.
    8 SubsidyAccounting built-in Tally up annual subsidy amounts required by barriers and strategies.

    Using the Framework

    To date, PlanZero posts have talked about critical success factors, barriers, and strategies informally. I will make an effort, going forward, to reflect at least some of these as code objects.

    Critical Success Factors

    A Critical Success Factors (CSF), as I've been using the term in the sector-by-sector NIR-reconstruction posts, can be formalized in terms of a triplet of (a) a sector, (b) an optimal value, and (c) a TimeSeries object. For example, a critical success factor such as "reduce the rate of methane emitted, per head" which is currently listed as a factor for Enteric Fermentation, can be formalized in this general way as follows:
    SectorMaximize / Minimize / MaintainTimeSeries KPI
    Enteric FermentationMinimize[average] methane / head
    A CSF does not generally suggest a particular mechanism of achieving an emissions reduction, it is just a callout as a potential indicator of progress: emission reductions in an IPCC-sector should occur if, and only if, there is progress on at least some of its CSFs. CSFs are closely related to Key Performance Indicators (KPIs); the TimeSeries used in a CSF definition can be called a KPI.

    Formalized CSFs in PlanZero are generally DynamicElement objects that also (additionally to what DynamicElement objects usually do)

    • name CSF TimeSeries variables
    • indicate whether each one should maximized, minimized, or maintained at a specific value
    • indicate the IPCC sector(s) to which each CSF applies
    Work remains to be done, but I hope eventually CSFs help visualize how strategies affect emissions, and generally, what's happening in a scenario.

    Barriers

    What I've been calling barriers in PlanZero posts to date typically correspond to constraints or challenges. The barriers currently associated Enteric Fermentation include, for example: "Owners of herds do not want to reduce their herds, and neither do their customers or suppliers."

    A formalized Barrier in PlanZero is a dynamic element that

    • is associated with any number (including zero) of IPCC sectors
    • reads strategy TimeSeries objects (see below) with non-zero delay
    • works with other Barrier DynamicElement objects to relate simulation inputs (e.g. StatsCan data regarding cattle headcounts) to simulation outputs (KPIs, e.g. annual sectoral {{CO2e|safe}} totals)
    I expect typical CSF and Strategy elements to have relatively simple logic. I expect Barrier elements to house the complexity. The strategy supporting this post is simply "use as much Bovaer as possible." It's not trivial to determine how much Bovear use is possible, and the logic for determining in the model, how much Bovaer it is possible to use in any given year, is in a Barrier element; it's fitting to call the element a "Barrier" because it isn't deemed possible to use enough for the entire national herd right away. Likewise, Bovaer usage doesn't immediately and completely eliminate methane; this frustrating fact supports again the word "Barrier" for the dynamic element that implements the modelling logic for how much methane is actually reduced, and at what cost.

    We'll see below the documentation on the Barrier elements in the scaling scenario, but taken together, they implement what I would call "the model" being simulated.

    Strategies

    Strategies are DynamicElement and TimeSeries objects that implement modelling assumptions, and serve to provide values for the TimeSeries objects that are the inputs to the CSF and Barrier DynamicElements.

    Not all Strategies are well-enough developed to be included in a Scenario. PlanZero includes Strategies that are quite speculative: all that's required to create a Strategy in PlanZero is a brief description of the idea, and a URL to learn more about it. In order to be eligible to participate in a scenario, a Strategy must be fully fledged as a DynamicElement: it must declare which TimeSeries variables it consults as inputs, and which ones it defines as outputs, and what logic it uses to relate them.

    A Scenario Is a data structure that comprises all of the TimeSeries objects, and all of the DynamicElement objects that make up a model of Canada's future, with regards to emissions.
